# Phase 13: 콘텐츠 고도화 (Content Refinement) - Research

**Researched:** 2026-07-22
**Domain:** Content quality — image relevance, markdown rendering, table formatting
**Confidence:** HIGH

## Summary

Phase 13 addresses three content quality problems in the mc pipeline: (R1) pollinations.ai images are keyword-based and lack contextual relevance; (R2) raw markdown symbols (`**`, `*`, `-`, `|`) leak into Hugo-rendered HTML; (R3) AI-generated markdown tables don't render properly in the Blowfish/PaperMod Hugo themes.

**Primary recommendation:** Attack all three with a layered approach. For R1, improve Pollinations prompt engineering (immediate win) and extend the existing Unsplash/Pexels real-photo providers (used only for thumbnails today) to body-image generation. For R2, add a dedicated markdown-sanitization step between `_extract_clean_body()` and `index.md` write that catches loose pipes/bold syntax. For R3, implement a Markdown→HTML table converter as a post-processing step since Hugo's Goldmark requires GFM-compliant tables and AI output is frequently malformed.

## R2: Markdown Symbol Leakage — Root Cause Analysis

### What Leaks, Where, and Why

| Symbol | Cause | Pipeline Stage |
|--------|-------|----------------|
| `\|` (pipe) | AI generates `\|` inside list text, inline code, or prose (not as a table). Hugo preserves literal `\|` in HTML. | `_publish_hugo()` → Hugo build |
| `**` (bold) | AI generates `**text**` but the spacing is incorrect (lack of surrounding whitespace), or bold syntax is inside code blocks/fenced areas. | AI generation → `draft_single_post()` |
| `*` (italic) | Same as `**` — spacing or code-block containment. | AI generation |
| `- ` (list) | Line starting with `-` inside inline code, or separator between paragraphs that Hugo renders as list-intended text. | `_extract_clean_body()` / Hugo build |
| `[](link)` | Malformed markdown links that Goldmark can't parse. | `_extract_clean_body()` |

### Primary Root Causes Identified

1. **AI generates `|` outside table context**: The prompt (`prompts.yaml` line 288) says "표를 사용했는가?" which encourages tables. But when the AI decides a table isn't suitable, it sometimes still includes `|` as column separators in running text, or writes pseudo-tables lacking the proper header-separator row.

2. **`_extract_clean_body()` does not clean markdown syntax**: The whitelist approach allows all markdown through. It strips HTML tags and JSON but **does not validate markdown correctness**. A malformed `| table` that Goldmark can't parse becomes literal `|` characters in HTML.

3. **Goldmark's strict parsing**: Hugo uses Goldmark, which is CommonMark-compliant and stricter than the AI's markdown output. It requires precise syntax (blank lines before block elements, correct table separators, proper bold spacing).

4. **AI prompt instructions about bold text**: `prompts.yaml` line 250: "볼드체 앞뒤에 반드시 공백 필요" — but AI sometimes generates `**text**` without proper CJK spacing, which Goldmark may not render as bold.

### Pipeline Points Where Leakage Enters

```
AI Output (raw) 
  → parse_ai_output() [removes JSON fences, HTML comments]
  → _strip_prompt_leak() [removes prompt headers, section blocks]
  → _extract_clean_body() [whitelist: allows all markdown through]
  → Hugo Goldmark [strict GFM parsing]
  → Rendered HTML [leaked symbols if markdown malformed]
```

**Fix location**: Between `_extract_clean_body()` and writing to `index.md`. Add a dedicated markdown validation/cleanup step.

### Fix Strategy

#### Step 1: Identify leakage (observability)
Add a logging step in `_publish_hugo()` after `_extract_clean_body()` that warns when markdown symbols (`|`, `**`, `*` at line start, `---` as horizontal rules) might leak. This makes the problem measurable.

#### Step 2: Add `_clean_markdown_symbols()` function
A new function that:
- Escapes `|` outside table context (within prose paragraphs, not in table rows)
- Validates bold/italic syntax has correct spacing
- Strips unmatched markdown delimiters
- Operates on the body only (preserves frontmatter)

#### Step 3: Fix AI prompt
In `prompts.yaml`, strengthen the rule about loose `|` characters. Add: "파이프(|) 문자는 반드시 표 안에서만 사용할 것. 본문 텍스트에서 | 문자를 사용하지 말 것."

#### Files to Modify
| File | Change |
|------|--------|
| `chain_publisher_core.py` | Add `_clean_markdown_symbols()` function; call after `_extract_clean_body()` in `_publish_hugo()` |
| `config/prompts.yaml` | Strengthen prompt rule about pipe characters |
| `test_chain_publisher_core.py` | Add tests for `_clean_markdown_symbols()` |

### Risk Assessment
| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Over-sanitization removes intentional `|` | Write regex that only touches `|` in non-table context |
| 112-test regression | LOW — new function is independent; existing tests unaffected |
| Hugo build errors from modified content | LOW — function operates on markdown before Hugo sees it |

## R3: Table Rendering — Root Cause Analysis

### Current State

- Hugo uses Goldmark parser which supports GFM tables by default (table extension is built-in, no config toggle needed) [CITED: github.com/yuin/goldmark]
- All three Hugo sites have `unsafe = true` for Goldmark renderer [VERIFIED: config files]
- Blowfish theme uses Tailwind CSS typography plugin (`prose`) which provides table styling at `main.css` lines 703-710: `width: 100%; table-layout: auto; margin-top: 2em; font-size: 0.875em;` [VERIFIED: code analysis]
- PaperMod has explicit `.post-content table th, .post-content table td` CSS at `post-single.css` lines 144-158 [VERIFIED: code analysis]
- Current published posts DO contain markdown tables (e.g., exam schedule tables in `/content/posts/`) [VERIFIED: grep on published content]

### Why Tables Break

Despite Goldmark supporting tables and the themes having table CSS, tables break for these reasons:

1. **Malformed AI tables**: The AI generates tables with inconsistent column counts. Example:
   ```
   | Header 1 | Header 2 |
   | ------- | ------- |
   | Cell 1 | Cell 2 | Extra cell |  ← extra column
   ```
   Goldmark treats this as three columns in the data row but two in the header — resulting in parsing failure. The markdown falls through as literal text with `|` characters.

2. **Missing blank line before table**: GFM requires a blank line before block elements. AI sometimes omits this:
   ```
   Some text
   | H1 | H2 |  ← no blank line → not a table
   ```

3. **Separator row not GFM-compliant**: Valid separators require at least one dash per column. AI sometimes uses:
   ```
   | H1 | H2 |
   | ---|---|  ← only 2 dashes — still valid in GFM, but edge cases fail
   ```

4. **Pipe `|` in cell content not escaped**: When table cell content contains a `|` (e.g., regex patterns, or/conditions), it creates extra columns.

5. **`_extract_clean_body()` table detection**: The function checks `re.match(r'^\||^[-|:\s]+$', stripped)` which correctly identifies table rows. However, if the table is not GFM-compliant, the `|` characters pass through the whitelist but Hugo can't render them as a table.

### Fix Options

#### Option A: Markdown table → HTML table conversion (Recommended)

| Aspect | Detail |
|--------|--------|
| **Effort** | MEDIUM (1-2 files) |

| **Why** | Hugo's `unsafe=true` setting preserves raw HTML. HTML tables are more robust and can be styled with the theme's existing CSS |
| **Parsing complexity** | Need to handle: header row, separator row (alignment), body rows, inconsistent column counts (pad with empty cells), escaped pipes |
| **File changes** | `chain_publisher_core.py` — add `_convert_tables_to_html()`, call in `_publish_hugo()` before `_extract_clean_body()` |

#### Option B: Table validation in schema gate (Supplemental)

| **Why** | Prevents bad tables from entering the pipeline, but doesn't fix existing published content |
| **Limitation** | Only catches issues; doesn't fix them |

#### Option C: pillow_chart table→image conversion (Not recommended for this phase)

## Common Pitfalls

### Pitfall 1: Table converter breaks on edge cases
**What goes wrong:** The table→HTML converter misparses complex tables (nested pipes in code, cells with colons in separator row, empty cells).
**Why it happens:** GFM table syntax has edge cases with alignment markers (`:---`, `:---:`), escaped pipes (`\|`), and empty cells (`||`).
**How to avoid:** Build test cases from actual AI output first. Use a regex-based parser that handles known patterns, not a full grammar.
**Warning signs:** Converter produces `<table>` with wrong column counts.

### Pitfall 2: Markdown cleaner too aggressive
**What goes wrong:** `_clean_markdown_symbols()` removes intentional `|` or `**` from prose content (e.g., programming tutorials with `|` operators).
**Why it happens:** Regex-based cleanup can't distinguish intentional from accidental.
**How to avoid:** Only clean inside markdown paragraphs that are not in code fences, not in tables (already detected), and not in link/image syntax. Use a whitelist context check.
**Warning signs:** Users report missing content in published posts.

### Pitfall 3: 112-test regression from `_extract_clean_body` changes
**What goes wrong:** Modifying `_extract_clean_body()` to be stricter breaks existing whitelist functionality.
**Why it happens:** The function uses a delicate whitelist balance — too restrictive, and valid content gets stripped.
**How to avoid:** Do NOT modify `_extract_clean_body()`. Add new functions (`_clean_markdown_symbols`, `_convert_tables_to_html`) that run AFTER the existing sanitizer.
**Warning signs:** Any of the 112 tests fail.

## Code Examples

### Table → HTML Converter Pattern

```python
# Source: Research synthesis for mc pipeline. Added to chain_publisher_core.py
# Requirements: (1) Handle GFM pipe tables, (2) Preserve alignment, (3) Pad inconsistent columns

import re
from typing import List, Tuple

def _convert_tables_to_html(body: str) -> str:
    """Convert GFM pipe tables in markdown body to HTML tables.
    
    Operates on body text only (frontmatter already extracted).
    Detects table blocks by looking for lines starting with | 
    preceded by a blank line.
    """
    lines = body.split('\n')
    result: List[str] = []
    i = 0
    
    while i < len(lines):
        line = lines[i]
        stripped = line.strip()
        
        # Detect table start: line starts with |, preceded by blank line (or start of body)
        if stripped.startswith('|') and (i == 0 or lines[i-1].strip() == ''):
            table_lines = []
            while i < len(lines) and lines[i].strip().startswith('|'):
                table_lines.append(lines[i].strip())
                i += 1
            
            html = _parse_table_to_html(table_lines)
            result.append(html)
            result.append('')
        else:
            result.append(line)
            i += 1
    
    return '\n'.join(result)


def _parse_table_to_html(table_lines: List[str]) -> str:
    """Parse GFM pipe table lines into <table> HTML."""
    if len(table_lines) < 2:
        return '\n'.join(table_lines)
    
    # Parse header row
    header_cells = _split_table_row(table_lines[0])
    
    # Parse separator row (detect alignment)
    sep_row = table_lines[1]
    alignments = _parse_alignment(sep_row)
    
    # Parse body rows
    body_rows = table_lines[2:]
    
    # Normalize column count to header
    col_count = len(header_cells)
    
    # Build HTML
    parts = ['<table>']
    
    # Header
    parts.append('  <thead>')
    parts.append('    <tr>')
    for i, cell in enumerate(header_cells):
        align = alignments[i] if i < len(alignments) else ''
        style = f' style="text-align: {align}"' if align else ''
        parts.append(f'      <th{style}>{cell.strip()}</th>')
    parts.append('    </tr>')
    parts.append('  </thead>')
    
    # Body
    if body_rows:
        parts.append('  <tbody>')
        for row in body_rows:
            cells = _split_table_row(row)
            # Pad or truncate to col_count
            while len(cells) < col_count:
                cells.append('')
            cells = cells[:col_count]
            
            parts.append('    <tr>')
            for cell in cells:
                parts.append(f'      <td>{cell.strip()}</td>')
            parts.append('    </tr>')
        parts.append('  </tbody>')
    
    parts.append('</table>')
    return '\n'.join(parts)


def _split_table_row(row: str) -> List[str]:
    """Split a GFM table row into cells, respecting escaped pipes."""
    row = row.strip()
    if row.startswith('|'):
        row = row[1:]
    if row.endswith('|'):
        row = row[:-1]
    
    cells = []
    current = []
    i = 0
    while i < len(row):
        ch = row[i]
        if ch == '\\' and i + 1 < len(row) and row[i + 1] == '|':
            current.append('|')
            i += 2
            continue
        if ch == '|':
            cells.append(''.join(current).strip())
            current = []
            i += 1
            continue
        current.append(ch)
        i += 1
    cells.append(''.join(current).strip())
    
    return cells


def _parse_alignment(sep_row: str) -> List[str]:
    """Parse separator row alignment markers (:---, :---:, ---:)."""
    cells = _split_table_row(sep_row)
    alignments = []
    for cell in cells:
        cell = cell.strip()
        if cell.startswith(':') and cell.endswith(':'):
            alignments.append('center')
        elif cell.endswith(':'):
            alignments.append('right')
        elif cell.startswith(':'):
            alignments.append('left')
        else:
            alignments.append('')
    return alignments
```

def build_contextual_prompt(

   - What's unclear: The exact malformations. We need to collect 3-5 actual drafts with table issues to validate the table converter regex.
   - Recommendation: Before implementing, run `grep -rn '|' output/drafts/` to collect actual table samples. Use these as test cases.

   - What we know: Hugo 0.134+ supports table render hooks.
   - What's unclear: Current Hugo version on this machine.
   - Recommendation: Check `hugo version` before deciding on render hooks approach.

3. **Should Unsplash/Pexels body images have text overlay?**
   - What we know: UnsplashProvider downloads real photos. Thumbnails add text overlay.
   - What's unclear: Body images should probably NOT have text overlay (unlike thumbnails which need title text).
   - Recommendation: Body images use raw provider download without `add_text_overlay`.
